# Все способы передачи инструкций в Claude Code CLI

**Claude Code CLI предоставляет многоуровневую систему инструкций**, где каждый метод занимает своё место в чёткой иерархии приоритетов. Главное, что нужно понять: **`--system-prompt` полностью заменяет системный промпт**, `--append-system-prompt` дополняет его, а файлы `CLAUDE.md` внедряются как пользовательское сообщение — не как часть системного промпта. Эта архитектурная разница определяет, какие инструкции «побеждают» при конфликтах. Ниже — полное сравнение всех методов с примерами, ограничениями и рекомендациями.

---

## Четыре флага системного промпта — ядро управления поведением

Claude Code предлагает четыре флага для управления системным промптом. Все работают как в интерактивном, так и в неинтерактивном (`-p`) режимах.

**`--system-prompt "текст"`** — **полностью заменяет** встроенный системный промпт Claude Code вашим текстом. Все стандартные инструкции — по работе с инструментами, стилю кода, правилам безопасности — удаляются. Это радикальная мера: вы получаете чистый лист, но теряете все встроенные возможности Claude Code. Используйте только когда нужен полный контроль, например, для узкоспециализированной автоматизации в CI/CD.

```bash
claude -p "проанализируй данные" --system-prompt "Ты дата-сайентист. Выполняй только статистический анализ."
```

**`--append-system-prompt "текст"`** — **дополняет** встроенный системный промпт вашим текстом в конце. Все стандартные возможности сохраняются. Официальная документация прямо рекомендует: **«Для большинства случаев используйте append-флаг»**. Это оптимальный выбор для добавления специфических правил сессии — фокуса на безопасности, стиля ответов, ограничений по технологиям.

```bash
# Код-ревью с фокусом на безопасность
claude -p "Проверь модуль аутентификации" \
  --append-system-prompt "Ищи SQL-инъекции, XSS, утечки токенов."

# Специфичная среда
claude --append-system-prompt "Работаем в WSL2. Используй 'service', а не 'systemctl'."
```

**`--system-prompt-file <путь>`** и **`--append-system-prompt-file <путь>`** — файловые варианты вышеуказанных флагов. Удобны для длинных промптов и воспроизводимости в пайплайнах.

```bash
claude --append-system-prompt-file ROLE.md --permission-mode acceptEdits "Начни реализацию"
```

Важные правила комбинирования: флаги `--system-prompt` и `--system-prompt-file` **взаимоисключающие** — нельзя использовать оба. Однако append-флаги **можно комбинировать** с любым из заменяющих флагов. В этом случае текст из append добавляется после заменяющего промпта.

| Флаг | Действие | Рекомендация |
|------|----------|--------------|
| `--system-prompt` | Заменяет весь системный промпт | Только для полного контроля |
| `--system-prompt-file` | Заменяет из файла | Для воспроизводимых пайплайнов |
| `--append-system-prompt` | Дополняет системный промпт | **Рекомендуется для большинства случаев** |
| `--append-system-prompt-file` | Дополняет из файла | Для длинных инструкций |

**Важный нюанс (GitHub issue #4523):** сообщество обнаружило, что `--append-system-prompt` может внедрять контент не строго в системный промпт API, а как пользовательское сообщение. Практическое следствие — в длинных сессиях добавленные инструкции могут «размываться» при компактизации контекста. Для критически важных постоянных правил лучше использовать `CLAUDE.md`.

## Позиционный аргумент и пайпинг через stdin

**Позиционный аргумент `[prompt]`** — это пользовательская задача или вопрос, который Claude будет выполнять. Он стоит на самом нижнем уровне приоритета — ниже системного промпта и CLAUDE.md. Без флага `-p` запускает интерактивную сессию; с `-p` — выполняет задачу и завершается.

```bash
# Интерактивная сессия с начальным промптом
claude "Объясни архитектуру этого проекта"

# Неинтерактивный режим
claude -p "Сгенерируй .gitignore для Node.js"

# Комбинация с другими флагами
claude --append-system-prompt "Отвечай в JSON" -p "Перечисли все функции в main.py"
```

**Пайпинг через stdin** (`echo "prompt" | claude -p`) позволяет передать контент из других команд как входной контекст. **Обязательно требует флаг `-p`** — без него интерактивный режим не получит TTY и выдаст ошибку. Пайп-контент и позиционный аргумент комбинируются: stdin даёт контекст, аргумент — инструкцию.

```bash
# Ревью git diff
git diff main | claude -p "Проверь изменения на проблемы безопасности"

# Анализ файла
cat myfile.php | claude -p "Найди баги в этом коде"

# Полный пайплайн с append-system-prompt
gh pr diff "$1" | claude -p \
  --append-system-prompt "Ты инженер по безопасности." \
  --output-format json

# Мониторинг логов
tail -f app.log | claude -p "Оповести, если увидишь аномалии"
```

**Ограничения stdin:** нельзя одновременно пайпить и работать в интерактивном режиме (feature request #6009 не реализован); при объёме stdin свыше ~7000 символов возможны пустые ответы (баг #7263); на headless-серверах могут быть проблемы с TTY даже в режиме `-p`.

## CLAUDE.md — постоянная память и правила проекта

Файлы `CLAUDE.md` — это **система постоянных инструкций**, которые Claude загружает автоматически в начале каждой сессии. Критически важный факт: **контент CLAUDE.md внедряется как первое пользовательское сообщение после системного промпта, а не как часть системного промпта**. Это означает, что `--append-system-prompt` имеет структурный приоритет над CLAUDE.md.

Файлы загружаются иерархически — от общих к специфичным:

| Расположение | Область | Загрузка | Для команды? |
|---|---|---|---|
| `~/.claude/CLAUDE.md` | Глобально | Всегда | Нет, персональный |
| `~/.claude/rules/*.md` | Глобальные правила | Всегда | Нет |
| `./CLAUDE.md` (корень проекта) | Проект | Всегда | Да, через git |
| `./.claude/CLAUDE.md` | Проект | Всегда | Да |
| `./.claude/rules/*.md` | Правила проекта | Всегда | Да |
| `./CLAUDE.local.md` | Персональный проект | Всегда | Нет, gitignored |
| `./subdir/CLAUDE.md` | Поддиректория | По запросу | Да |

**Принцип приоритета:** более специфичные файлы «побеждают» более общие при конфликтах, но все уровни **комбинируются** (не заменяют друг друга). Файлы правил в `.claude/rules/` поддерживают таргетирование по путям через YAML-фронтматтер:

```yaml
---
paths:
  - "src/api/**/*.ts"
---
# Правила разработки API
- Все эндпоинты должны включать валидацию входных данных
```

Поддерживается синтаксис `@path/to/file` для модульного импорта, рекурсивно до 5 уровней. Команда `/init` создаёт стартовый CLAUDE.md на основе структуры проекта. **Рекомендуемый объём — до 200 строк**: при превышении качество следования инструкциям снижается. Для усиления соблюдения директив — объясняйте **почему** правило важно, а не только что нужно делать.

## Флаг `--agent` и система агентов

**`--agent <имя>`** загружает конфигурацию пользовательского агента как основной режим сессии. В отличие от субагентов, которые Claude может вызывать по ходу работы, `--agent` превращает **всю сессию** в работу специализированного агента с его моделью, набором инструментов и правилами.

Агенты определяются как Markdown-файлы с YAML-фронтматтер в `.claude/agents/` (проектный уровень) или `~/.claude/agents/` (глобальный уровень):

```markdown
---
name: security-reviewer
description: Аудит безопасности кодовой базы
model: opus
tools: Read, Grep, Glob
permissionMode: plan
---
Ты аудитор безопасности. Фокусируйся на:
- Уязвимости аутентификации
- Проблемы валидации входных данных
- Утечки секретов
Никогда не модифицируй файлы напрямую.
```

```bash
claude --agent security-reviewer "Проведи аудит этого кодабазы"
```

Отдельно существует **`--agents` (множественное число)** — для определения субагентов через JSON прямо в командной строке. Эти субагенты доступны главному агенту для делегирования задач во время сессии, но не становятся основным режимом:

```bash
claude --agents '{
  "code-reviewer": {
    "description": "Эксперт по код-ревью",
    "prompt": "Ты старший ревьюер. Фокусируйся на качестве и безопасности.",
    "tools": ["Read", "Grep", "Glob", "Bash"],
    "model": "sonnet"
  }
}'
```

Флаг `--agent` переопределяет настройку `agent` в `settings.json`. Проектные агенты переопределяют глобальные с тем же именем.

## settings.json, переменные окружения и MCP

**`.claude/settings.json`** управляет разрешениями, переменными окружения, хуками и поведением инструментов. Файл **не содержит промпт-текст напрямую** — для инструкций используйте CLAUDE.md. Иерархия настроек: **Managed > Project > User > Local**. Массивы (например, `permissions.allow`) конкатенируются; скалярные значения — побеждает более приоритетный уровень.

```json
{
  "permissions": {
    "allow": ["Bash(npm run lint)", "Bash(npm run test *)"],
    "deny": ["Read(./.env)", "Read(./.env.*)"]
  },
  "env": { "ANTHROPIC_MODEL": "claude-sonnet-4-6" },
  "hooks": {
    "PostToolUse": [{
      "matcher": "Write(*.py)",
      "hooks": [{ "type": "command", "command": "python -m black \"$file\"" }]
    }]
  }
}
```

**Ключевые переменные окружения** управляют поведением на системном уровне. CLI-аргументы переопределяют переменные окружения, которые в свою очередь переопределяют значения `env` в settings.json.

| Переменная | Назначение |
|---|---|
| `ANTHROPIC_MODEL` | Модель по умолчанию |
| `CLAUDE_CODE_MAX_OUTPUT_TOKENS` | Максимум токенов ответа |
| `CLAUDE_CODE_SUBAGENT_MODEL` | Модель для субагентов |
| `MAX_THINKING_TOKENS` | Токены для «мышления» модели |
| `BASH_DEFAULT_TIMEOUT_MS` | Таймаут bash-команд |
| `MCP_TIMEOUT` / `MCP_TOOL_TIMEOUT` | Таймауты MCP-серверов |

**MCP-конфигурация** расширяет Claude Code внешними инструментами (базы данных, API, браузер). MCP-серверы не инжектируют промпт-текст, но добавляют описания инструментов в контекст каждого запроса. Конфигурация хранится в `.mcp.json` (проект), `~/.claude.json` (пользователь), или управляется через `claude mcp add/remove/list`.

```bash
# Добавить HTTP MCP-сервер
claude mcp add --transport http github https://api.githubcopilot.com/mcp/

# Добавить stdio-сервер
claude mcp add --transport stdio postgres -- npx -y @modelcontextprotocol/server-postgres "$DATABASE_URL"
```

## Полная иерархия приоритетов и как методы взаимодействуют

Вот **итоговая иерархия** от наивысшего к низшему приоритету — это ключевая модель для понимания того, какие инструкции «победят» при конфликтах:

- **Managed settings** (серверные/MDM-политики) — не могут быть переопределены
- **`--system-prompt`** — полная замена системного промпта
- **`--append-system-prompt`** — дополнение системного промпта (или заменяющего промпта, если `--system-prompt` тоже указан)
- **Output styles** (`.claude/output-styles/`) — часть системного промпта
- **CLAUDE.md файлы** — внедряются как первое пользовательское сообщение
- **Auto memory** (`MEMORY.md`, первые 200 строк)
- **Пользовательский промпт** (позиционный аргумент / stdin) — самый низкий приоритет

Практический пример взаимодействия: если `CLAUDE.md` говорит «Пиши на Python 3.10», а `--append-system-prompt` указывает «Используй Python 3.12» — победит `--append-system-prompt`, потому что системный промпт имеет структурный приоритет над пользовательскими сообщениями. Если же конфликтуют `CLAUDE.md` на уровне проекта и позиционный аргумент пользователя — `CLAUDE.md` приоритетнее, поскольку Claude Code трактует его как авторитетные системные правила, а пользовательский ввод — как гибкий запрос в рамках этих правил.

## Заключение

Система инструкций Claude Code CLI спроектирована как многослойная архитектура: **системный промпт задаёт фундаментальное поведение**, CLAUDE.md обеспечивает постоянный проектный контекст, а позиционные аргументы и stdin — разовые задачи. Самая распространённая ошибка — считать, что `--append-system-prompt` и `CLAUDE.md` работают одинаково; на деле они действуют через разные механизмы с разным приоритетом. Для повседневной работы `--append-system-prompt` в сочетании с хорошо написанным `CLAUDE.md` покрывает **95% сценариев**. Флаг `--system-prompt` нужен только при полной кастомизации для автоматизации. Агенты (`--agent`) — для специализированных сессий с изменённой моделью, набором инструментов или правами доступа. А переменные окружения и `settings.json` управляют инфраструктурным поведением, не затрагивая промпт-контент напрямую.
